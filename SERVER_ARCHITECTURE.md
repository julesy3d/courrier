# Yeet — Server Architecture

> **Source of truth.** If something contradicts this document, this document wins.

## Overview

Yeet is a head-to-head video duel app. The entire backend runs on **Supabase** (Postgres + Auth + Storage). All mutations go through **RPC functions** (plpgsql, `SECURITY DEFINER`). The client never writes directly to tables except `comments`.

**Supabase project:** `https://kevfztwwnioouohjmaoy.supabase.co`

## Fuel System (`pending_views`)

The core distribution mechanic. Every card has a `pending_views` counter — its "fuel."

### How fuel enters the system
- **New video created:** `pending_views = 7` (guaranteed minimum exposure)
- **Win at streak position k:** `pending_views += ceil((k + 1) / 2)`

| Streak position | Fuel earned | Cumulative after K wins |
|-----------------|-------------|------------------------|
| 1st | +1 | 1 |
| 2nd | +2 | 3 |
| 3rd | +2 | 5 |
| 4th | +3 | 8 |
| 5th | +3 | 11 |
| 10th | +6 | 33 |

### How fuel leaves the system
- **Card placed in a matchup:** `pending_views -= 1` per card. Both cards in a non-ladder matchup spend 1 each. In a ladder matchup, only the challenger spends 1 (the defender is already on screen).
- **Card hits 0:** stops appearing as an opponent. Can still defend in active ladders.

### How fuel is recovered
- **Stale matchup expires** (unjudged > 24h): both cards get `pending_views += 1` refund.

### Distribution model
**Pull-based, not push-based.** There is no fan-out after a win. The winner earns fuel (pending_views), and that fuel is spent when ANY user needs a matchup. This is controlled by `generate_matchups_for_user`.

## RPC Functions

### `create_card(p_video_url text) → uuid`
Creates a new post for the authenticated user.
- Sets `pending_views = 7`
- Returns the new post UUID
- Does NOT generate any matchups (old `generate_matchups` call removed)

### `generate_matchups_for_user(p_user_id uuid, p_count integer) → integer`
**Pull-based matchup generator.** Creates up to `p_count` matchups for a user.

For each matchup:
1. Pick card A: random, `pending_views > 0`, not by this user
2. Pick card B: random, `pending_views > 0`, not by this user, different creator than A, **never seen as a pair by this user** (no-rematch rule)
3. Insert matchup row
4. Both cards: `pending_views -= 1`

Returns count of matchups actually created (may be less than requested if not enough eligible cards).

**No-rematch check:** queries ALL matchups (judged AND unjudged) for this viewer to ensure the pair `(A, B)` or `(B, A)` has never been shown.

### `sync_matchups(p_after timestamptz DEFAULT NULL) → jsonb`
Main client sync endpoint. Returns unjudged matchups as a JSON array.

1. Count user's unjudged matchups
2. If fewer than **5** (magazine size), call `generate_matchups_for_user` to top up
3. Return all unjudged matchups (optionally filtered by `p_after` high-water mark)

Each returned matchup includes full card data:
```json
{
  "matchup_id": "uuid",
  "card_a_id": "uuid",
  "card_b_id": "uuid",
  "created_at": "timestamp",
  "card_a": {
    "id": "uuid",
    "video_url": "https://...",
    "creator_username": "display_name",
    "emoji_tallies": {},
    "total_wins": 0,
    "comment_count": 0
  },
  "card_b": { ... }
}
```

Returns `'[]'::jsonb` when no matchups available.

### `judge_matchup(p_matchup_id uuid, p_kept_card_id uuid, p_emoji text DEFAULT NULL, p_streak integer DEFAULT 1) → jsonb`
Core judgment function. Called when user swipes.

**Validations:**
1. User exists (via `auth.uid()`)
2. Emoji is valid or null
3. Matchup exists, belongs to this user, not already judged (`FOR UPDATE` lock)
4. Kept card is part of this matchup

**Actions:**
1. Mark matchup judged (`kept_card_id`, `killed_card_id`, `emoji`, `judged_at`)
2. Winner: `total_wins += 1`
3. Winner: `pending_views += ceil((p_streak + 1) / 2.0)`
4. If emoji provided: increment `emoji_tallies[emoji]`
5. Insert `reposts` row (journey log)
6. Killed card: **no penalty.** It already spent 1 `pending_views` when the matchup was created. Losing IS the cost.
7. **Ladder:** pick one random opponent where:
   - Not the winner or loser
   - `pending_views > 0`
   - Not created by this user
   - Has `video_url`
   - This user has **never seen** winner vs this opponent (full no-rematch check)
8. If opponent found: insert ladder matchup (`card_a = winner`, `card_b = opponent`), decrement opponent's `pending_views` by 1

**Returns:**
```json
{
  "kept": "uuid",
  "killed": "uuid",
  "emoji": null,
  "streak": 1,
  "streak_bonus": 1,
  "kept_total_wins": 5,
  "kept_pending_views": 12
}
```

### `backfill_new_user_v2(p_user_id uuid) → integer`
Seeds a new user with matchups from high-scoring cards. Reads `backfill_count` from `config` table (currently 5). Both cards in each matchup spend 1 `pending_views`.

### `expire_stale_matchups() → integer`
**Cron job.** Finds unjudged matchups older than 24 hours, refunds both cards 1 `pending_views` each, deletes the matchup row. Uses `FOR UPDATE SKIP LOCKED` for safety.

Scheduled daily at 3am UTC via `pg_cron`:
```sql
SELECT cron.schedule('expire-stale-matchups', '0 3 * * *', $$SELECT expire_stale_matchups()$$);
```

### `get_leaderboard(p_limit integer DEFAULT 20) → jsonb`
Returns top cards ranked by `total_wins`. Used by the leaderboard screen.

Returns: `[{ id, video_url, total_wins, pending_views, creator_username, rank }]`

### `heartbeat() → void`
Updates `users.last_active_at` to `now()`.

### `compute_daily_tops() → void`
Cron (00:05 UTC). Finds top card per emoji for yesterday. Writes to `daily_tops`.

## Client ↔ Server Contract

### Sync flow
```
App opens / resumes
  → store.syncMatchups()
    → RPC sync_matchups(p_after: highWaterMark)
      → Server auto-fills magazine if < 5 pending
    → Client deduplicates, appends to cachedMatchups
```

### Judgment flow
```
User swipes (yeet gesture)
  → Client tracks streak locally (ref counter)
  → store.judgeMatchup(matchupId, keptCardId, null, streak)
    → Optimistic: remove from cachedMatchups IMMEDIATELY
    → RPC judge_matchup(matchupId, keptCardId, null, streak)
      → Server: score, fuel, ladder matchup created
  → store.syncMatchups()
    → Returns new matchups including ladder matchup
  → Client finds ladder matchup (card_a_id === keptCardId)
    → Swap dead slot, animate challenger in
    → If no ladder found: load next independent matchup
```

### Streak tracking
- Client-side only (refs in MatchupView)
- Incremented when the same card wins consecutively
- Reset to 1 when: card loses, no ladder matchup found, app backgrounds
- Passed to `judge_matchup` as `p_streak` parameter
- Server trusts the value (low stakes — gaming it just means your video gets shown more)

### Card creation flow
```
User records video (5s, hold-to-record)
  → Upload MP4 to card_videos bucket
  → store.createCard(videoUrl)
    → RPC create_card(videoUrl)
      → Insert post with pending_views = 7
      → No immediate matchup generation
    → Video enters the pool, gets picked when users need matchups
```

## Invariants

These must always hold. If code violates any of these, it's a bug.

1. **A card with `pending_views = 0` is never placed in a new matchup.** It can still appear in existing unjudged matchups.
2. **The same pair (A, B) is never shown to the same viewer twice**, regardless of order.
3. **A user never judges their own card.** Both `generate_matchups_for_user` and the ladder query exclude `sender_id = viewer`.
4. **A user never sees two cards by the same creator in one matchup.** Enforced by `generate_matchups_for_user`.
5. **`total_wins` only increments by 1 per judgment.** The streak bonus only affects `pending_views`, not score.
6. **Optimistic cache removal happens BEFORE the RPC call.** The animation must complete before `judgeMatchup` is called, or the component unmounts mid-animation.
7. **In ladder matchups, `card_a` is always the defender.** The client uses `card_a_id === keptCardId` to identify ladder matchups.
8. **`pending_views` never goes below 0.** All decrements use `GREATEST(pending_views - 1, 0)`.

## Row Level Security (RLS)

All tables have RLS enabled. All RPC functions are `SECURITY DEFINER` and bypass RLS entirely — these policies only gate **direct PostgREST access** (i.e. `supabase.from('table')` calls from the client).

### Policies

| Table | SELECT | INSERT | UPDATE | DELETE |
|-------|--------|--------|--------|--------|
| `users` | All authenticated | Own row (`auth_id = auth.uid()`) | Own row (`auth_id = auth.uid()`) | — |
| `posts` | All authenticated | — (via RPC) | — (via RPC) | — |
| `matchups` | Own only (`viewer_id` = current user) | — (via RPC) | — (via RPC) | — |
| `comments` | All authenticated | Own (`author_id` = current user) | — | — |
| `reposts` | All authenticated | — (via RPC) | — | — |
| `daily_tops` | All authenticated | — (via RPC) | — | — |
| `config` | All authenticated | — | — | — |

### Why these policies exist

The client makes direct table calls in a few places:
- **`users`** — SELECT to load current user by `auth_id`, check username availability; INSERT during onboarding; UPDATE for language change
- **`posts`** — SELECT to fetch own outbox (`sender_id = currentUser.id`)
- **`matchups`** — SELECT count of own unjudged matchups on home screen
- **`comments`** — SELECT for post log; INSERT for adding comments
- **`reposts`** — SELECT with join to `users` for post log

Everything else (creating cards, judging matchups, generating matchups, backfill, leaderboard, heartbeat) goes through `SECURITY DEFINER` RPCs and is unaffected by RLS.

## What's NOT in the server

- **Feed / discovery / algorithm** — There is none. Content spreads only through the duel mechanic.
- **Push notifications for matchups** — Not implemented. Users see matchups when they open the app.
- **Video transcoding** — Videos are uploaded as-is from the device camera (H.264, varying resolution).
- **Moderation** — None yet.
- **Daily contest winner announcement** — `compute_daily_tops` writes to `daily_tops` but nothing reads it yet.
